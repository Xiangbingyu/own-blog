 # Redis 在认证流程中的关键作用：Token 生命周期与权限缓存策略

在基于 JWT 的登录体系里，很多团队都会遇到一个经典问题：Token 本身是无状态的，只要没有过期，服务端就会认为它有效。这样一来，即使用户已经退出登录，旧 Token 仍可能在过期前继续访问接口。

要解决这个问题，核心并不是放弃 JWT，而是给它补上“服务端状态感知”。Redis 正好适合承担这层职责。

## 一、为什么 Redis 是认证链路里的关键组件

Redis 在认证流程中的价值可以概括为三点：

- 记录 Token 生命周期，让“退出登录立即失效”可控
- 降低权限查询对 SQL 的压力，提升鉴权吞吐
- 用 TTL 把“安全规则”自动化，不依赖人工清理

也就是说，JWT 负责“自包含身份信息”，Redis 负责“实时状态与策略控制”，两者是互补关系。

## 二、Token 管理：白名单与黑名单结合

### 1）白名单：只信任已签发且仍有效的 Token

在登录或刷新 Token 时，把 Token 的唯一标识（如 `jti`）写入 Redis 白名单：

```text
key: auth:token:whitelist:{jti}
value: userId / deviceId / sessionVersion
ttl: token 剩余有效期
```

请求到来时，先校验 JWT 签名与 exp，再查白名单。查不到就拒绝访问。这样可以避免“伪造结构合法但未被系统签发”的 Token 混入。

### 2）黑名单：处理主动失效场景

用户退出登录、账号被踢下线、检测到风险行为时，把当前 Token 放入黑名单：

```text
key: auth:token:blacklist:{jti}
value: revokeReason
ttl: token 剩余有效期
```

鉴权顺序通常是：先查黑名单，命中即拒绝；再查白名单，未命中也拒绝。这样可以保证“退出后立刻失效”，不再依赖客户端是否删除本地 Token。

### 3）为什么要白名单 + 黑名单同时存在

只用黑名单，会默认信任所有未被拉黑的 Token，控制粒度偏粗。  
只用白名单，能做强校验，但处理“主动失效”时不够直观。  
两者组合后，既能控制签发范围，又能快速撤销访问权。

## 三、TTL：让 Token 生命周期自动闭环

Redis 的 TTL 是这套方案的关键：

- 白名单 TTL = Token 剩余有效期，过期自动删除
- 黑名单 TTL = Token 剩余有效期，避免永久占用内存
- 刷新 Token 轮换时，旧 Token 可直接入黑名单并继承剩余 TTL

这样做的好处是，认证状态会随时间自然收敛，不需要额外定时任务清理脏数据，逻辑也和安全策略保持一致。

## 四、权限查询也放 Redis：从“鉴权正确”到“鉴权高并发”

很多系统在接口鉴权时会频繁查数据库判断权限，例如：

- 用户是否有某个菜单权限
- 用户是否具备某个操作权限（如 `order:refund`）
- 用户所在角色是否被禁用

如果每次请求都访问 SQL，QPS 上来后数据库会先成为瓶颈。更合理的做法是把权限结果缓存到 Redis。

### 1）正向缓存：有权限结果直接缓存

```text
key: auth:perm:user:{userId}
value: 权限集合（set/json）
ttl: 5~30 分钟（按业务变更频率设定）
```

命中 Redis 就直接判定，无需落库查询。

### 2）负向缓存：无权限结果也缓存

这是很多人会忽略的一点：当 Redis 没有命中且 SQL 也判定“不允许”时，也应该把这个“拒绝结果”写入 Redis，避免穿透。

例如：

```text
key: auth:perm:deny:user:{userId}:action:{action}
value: 1
ttl: 60~300 秒
```

下次同类请求可以快速拒绝，不必每次都打到 SQL。这个策略在“恶意探测权限”和“高频无权限请求”场景下效果非常明显。

## 五、一条可落地的鉴权流程

可以按下面的顺序执行认证与授权：

1. 解析并校验 JWT（签名、过期时间、基础声明）
2. 检查黑名单（命中即拒绝）
3. 检查白名单（未命中即拒绝）
4. 查询权限缓存（命中直接放行/拒绝）
5. 缓存未命中时查 SQL
6. 将 SQL 结果写回 Redis（包含允许与不允许两类结果）

这条链路同时覆盖了安全性、实时失效能力和性能稳定性。

## 六、实践建议

- Token 建议携带 `jti`，避免直接把完整 Token 作为 Redis key
- 黑名单 TTL 必须精确到“剩余有效期”，不要写死超长时间
- 权限缓存要有版本号或失效机制，角色变更时可主动删除相关 key
- 负向缓存 TTL 不宜过长，通常用短 TTL 防止权限刚变更后体验异常

## 小结

Redis 在认证流程里并不是“可选优化”，而是把 JWT 方案工程化、可控化的关键一环。  
通过白名单 + 黑名单 + TTL，你可以真正管理 Token 的生命周期，彻底规避“退出后仍可访问”的风险。  
再结合权限正向缓存与负向缓存，可以在保证鉴权正确性的同时，显著降低 SQL 压力，让系统在高并发下依然稳定。
